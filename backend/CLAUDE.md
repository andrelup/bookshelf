# BookShelf Backend — Hexagonal Architecture

REST API for a bookstore. This CLAUDE.md complements the monorepo root CLAUDE.md.

## Stack

- Python 3.12
- FastAPI (async)
- SQLAlchemy 2.0 (async, declarative models)
- Alembic (autogenerated migrations)
- PostgreSQL 16 + pgvector (semantic search)
- pytest + pytest-asyncio + httpx (testing)
- structlog (structured logging)
- Ruff (linting + formatting)
- mypy strict (type checking)

## Hexagonal Architecture (Ports & Adapters)

## Hexagonal Architecture rules

1. **The domain does NOT import from adapters.** Ever. Imports always flow inward:
   - `adapters.inbound.api` → imports from `domain.services` and `domain.models`
   - `adapters.outbound.persistence` → imports from `domain.ports` and implements its interfaces
   - `domain.services` → imports from `domain.ports` and `domain.models`, NOTHING else

2. **Ports are Protocol classes** (typing.Protocol), not abstract classes:
   ```python
   from typing import Protocol

   class BookRepository(Protocol):
       async def find_by_id(self, book_id: int) -> Book | None: ...
       async def find_all(self, skip: int, limit: int) -> list[Book]: ...
       async def save(self, book: Book) -> Book: ...
       async def delete(self, book_id: int) -> None: ...
   ```

3. **Domain models are independent of SQLAlchemy.** Use dataclasses or Pydantic BaseModel for the domain. SQLAlchemy models live in `adapters/outbound/persistence/sqlalchemy_models.py` and are mapped to and from the domain models.

4. **Dependency injection in `config/container.py`.** It wires the ports to their concrete implementations. Routers receive the domain services already injected via FastAPI Depends.

5. **Domain exceptions are translated in the error_handler middleware.** `BookNotFoundError` → HTTP 404, `UnauthorizedError` → HTTP 401, `ValidationError` → HTTP 422.

## Code conventions

- snake_case for everything (variables, functions, files, modules)
- PascalCase only for classes (models, schemas, exceptions, protocols)
- Async/await in every endpoint, service and repository
- Type hints mandatory on all functions (mypy strict)
- Docstrings on domain services and complex functions
- API responses always in this format:
  ```json
  {"success": true, "data": {...}, "error": null}
  {"success": false, "data": null, "error": "Book not found"}
  ```
- Pydantic schemas split per operation: `BookCreate`, `BookUpdate`, `BookResponse`, `BookListResponse`

## Testing

- Framework: pytest + pytest-asyncio + httpx.AsyncClient
- AAA pattern: Arrange → Act → Assert (with comments)
- Fixtures centralized in `conftest.py`: test DB, async client, authenticated user (customer and seller)
- Unit tests: mock the ports, test domain services in isolation
- Integration tests: use a real test DB, verify the repositories
- API tests: httpx.AsyncClient against the FastAPI app
- Minimum coverage: 80%
- Run: `pytest --cov=src --cov-report=term-missing`

## Migrations

- Alembic with autogenerate from the SQLAlchemy models (not from the domain models)
- Create a migration: `alembic revision --autogenerate -m "description"`
- Apply: `alembic upgrade head`
- Rollback: `alembic downgrade -1`
- Always review the generated SQL before applying it

## Logging

- structlog with JSON in production, pretty-print in development
- Automatic fields: timestamp, request_id, user_id, endpoint, method, status_code, duration_ms
- Never use `print()` — always `logger.info()`, `logger.error()`, etc.

## Linting

- Ruff as the single linter and formatter
- mypy in strict mode
- Pre-commit hooks: ruff check, ruff format, mypy

## What NOT to do

- Do not import SQLAlchemy, FastAPI, httpx or any infrastructure library inside `domain/`
- Do not put business logic in the routers — routers only validate input, call the service, and return the response
- Do not use `from sqlalchemy import ...` in `domain/services/`
- Do not create endpoints without their corresponding Pydantic schema
- Do not write direct SQL queries — always go through the repository
